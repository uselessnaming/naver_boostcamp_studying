# NoSQL
Not Only SQL을 의미하며, NoSQL의 경우 SQL 뿐만 아니라 자체적인 문법을 제공한다.     
NoSQL의 가장 큰 장점은 유연함이다.      
어떤 데이터의 형식이든 자유롭게 저장이 가능하다. 그리고 확장 및 수정이 용이하다.        
또한 대량의 데이터를 읽어오고 처리하기에 적합하다.

## 비정형 데이터 저장
데이터의 형식의 제한이 없기 때문에 자유롭게 처리할 수 있다.     
예를 들어, 상품 데이터를 저장하고자 할 때 의류와 식품을 저장하면 추가적인 정보들이 다르기도 하다.       
이런 경우 RDB의 경우는 각각을 별도의 테이블로 구분하여 도메인 정의를 새로 하거나 Nullable하게 하여 불필요한 값도 관리해야 한다.     
따라서 이런 경우 NoSQL이 장점을 가지기도 한다.

## 확장 및 수정
데이터 테이블을 구성하고 변경해야 할 때 RDB의 경우에 ALTER를 활용하여 변경할 수 있다.       
하지만 제약 조건 등등 관리할 부분들이 존재하기 때문에 복잡한 편이다.        
이는 관계를 기반으로 하기 때문이다.     
그에 비해 NoSQL의 경우에는 별도의 제약이 크게 없기 때문에 바로 수정할 수 있다.

## 대량의 데이터
RDB에서는 데이터를 읽어올 때 JOIN이 필요한 경우가 많다. 
이 때 데이터가 단순 10000단위가 아니라 100000, 혹은 그 이상의 큰 데이터를 관리하게 된다면 JOIN 자체에 부담이 되는 경우가 발생한다.      
이에 비해서 NoSQL의 경우에는 데이터를 보유하고 있기 때문에 속도 측면에서 우위를 가질 수 있다

# RDB
관계형 데이터 베이스를 말하며, RDB의 경우에는 대상이 되는 Entity를 정의하고 이들 간의 관계를 정의하여 관리한다.     
JOIN을 통해 여러 데이터베이스에서 데이터를 가져올 수 있다.      
테이블에서 하나의 데이터를 관리하고 관계를 별도의 테이블로 관리하기 때문에 최소한으로 필요한 데이터만 보유할 수 있다.       
즉, 공간의 효율성이 늘어난다.       
그리고 스키마가 기존에 정의되어 있기 때문에 형식이 명확하게 분리된다.     

## 공간의 효율성
NoSQL은 결국 JOIN이라는 연산이 존재하지 않기 때문에 데이터를 중복적으로 보유할 수 밖에 없다.        
예를 들어, 학생이 수강 신청을 한다고 생각하면 NoSQL에서는 학생 데이터에 수강 신청 정보를 포함시켜야 하고, 별도의 수강 신청 정보에도 학생의 정보가 일부 들어가게 된다.       
그에 비해 RDB로 관리하게 되면 학생의 정보, 강의 정보를 테이블로 구성한다. 그리고 이 정보들을 PK를 가지고 관계를 맺어 하나의 수강신청 테이블을 관리하게 된다. 결과적으로 데이터의 중복이 사라지게 된다.      

## 명확한 스키마
NoSQL은 스키마가 존재하지 않는다. 비정형 데이터를 관리하기 때문에 명확한 스키마를 정의할 수 없다.       
물론 사람이 정의해두면 훨씬 편하겠지만, 제약은 크게 존재하지 않는다.        
RDB는 스키마가 존재하며, 이를 정의해두면 도메인과 맞지 않는 데이터는 삽입이 불가하다.       
따라서 RDB에서는 명확한 스키마로 인해 데이터가 명확하게 정의될 수 있다.     

# RDB vs NoSQL
대규모의 데이터를 관리하거나 데이터베이스가 변경될 경우가 자주 있다면 NoSQL을 활용하는 것이 좋다.       
데이터를 효율적으로 저장하고 관리하며, 엄격한 제약을 통해서 데이터를 관리할 수 있다.
